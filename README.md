### 2. Відповіді на теоретичні питання
#### Питання 1: Чому аудіо-характеристики винесені в окремий об’єкт audio_features, а не зберігаються плоско? Коли таке вкладення вигідне, а коли створює проблеми?
Обгрунтування: Це патерн логічного групування (Logical Grouping). Аудіо-фічі (темп, акустичність, енергійність) описують суто фізичні властивості звукової доріжки і суттєво відрізняються від загальних метаданих треку (назва, жанр, альбом).

Коли це вигідно:
Оптимізація вибірки (Проєкція): Якщо для інтерфейсу застосунку потрібні лише базові дані, можна легко виключити весь блок фіч за допомогою { audio_features: 0 }, зменшуючи навантаження на мережу та RAM.

Зручність для аналітики/ML: При передачі даних у моделі рекомендаційних систем зручніше оперувати одним готовим об'єктом, ніж збирати 12 розрізнених полів по всьому документу.

Коли це створює проблеми:
Громіздкі запити: Для пошуку та індексації доводиться постійно писати довгі шляхи через крапкову нотацію, наприклад: db.tracks.find({ "audio_features.tempo": { $gt: 120 } }).

Ризик затирання даних: При оновленні вкладеного об'єкта без використання точкового оператора $set можна випадково видалити інші характеристики, що містилися всередині об'єкта.

#### Питання 2: Чому виконавці зберігаються як масив, а не як рядок? Які запити стають простішими?
Обгрунтування: Рядок типу "Artist A; Artist B" порушує першу нормальну форму (1NF), оскільки дані в межах одного поля не є атомарними. Масив — це природний NoSQL-спосіб реалізації зв'язків один-до-багатьох (1:N).

Які запити стають простішими:

Точний пошук: Пошук треків конкретного виконавця виконується простим запитом db.tracks.find({ artists: "Eminem" }). Немає потреби у важких регулярних виразах ($regex), які змушують базу робити повне сканування колекції (Collection Scan).

Пошук спільних треків (коллаборацій): Запити на перетин множин стають тривіальними завдяки оператору $all: db.tracks.find({ artists: { $all: ["Jay-Z", "Linkin Park"] } }).

Індексація (Multikey Index): MongoDB дозволяє побудувати індекс по масиву. База створить окремий покажчик для кожного артиста, що робить пошук за іменем миттєвим. З текстовим рядком така індексація працювала б набагато повільніше та менш ефективно.

#### Питання 3: Що таке $out і чим він відрізняється від $merge? Коли використовувати кожен?
Обидва оператори використовуються на фінальній стадії конвеєра агрегації (Aggregation Pipeline) для збереження результатів, але вони мають кардинально різну логіку роботи під капотом:

**$out (Повний перезапис):**

Повністю знищує (робить drop) цільову колекцію, якщо вона існувала, і створює її з нуля, записуючи туди нові документи.

Ідеально підходить для пакетної обробки (Batch Processing), перегенерації щомісячних/щоденних звітів, матеріалізованих представлень, або під час первинної міграції та повної трансформації даних (як у поточному завданні).

**$merge (Інкрементальне оновлення):**

Не видаляє існуючу колекцію. Він зіставляє документи з результату агрегації з документами в цільовій колекції за унікальним ключем (наприклад, _id). Залежно від умов, він може оновити старий документ, додати новий (якщо збігів не знайдено) або виконати гнучку логіку злиття полів.

Незамінний для обробки даних у реальному часі. Наприклад, коли потрібно додати нові щоденні прослуховування до вже існуючої історичної бази статистики за рік, не перераховуючи при цьому дані за попередні 11 місяців.

### Для чого використовується інструкція `$unwind`?
Інструкція `$unwind` використовується для денормалізації(розгортання) масивів у документах під час виконання конвеєра агрегації(Aggregation Pipeline). 

Коли MongoDB зустрічає поле, яке є масивом, вона не може застосувати до його елементів стандартні операції групування або фільтрації так, ніби це окремі документи. Етап `$unwind` бере один вихідний документ, що містить масив із `N` елементів, і розбиває його на `N` окремих копій-документів. У кожній такій копії на місці колишнього масиву тепер міститься лише один конкретний його елемент.

* **Коли це критично необхідно:**
  * **Групування та аналітика:** Як у завданні з артистами, де один трек міг належати масиву з трьох виконавців (`["Eminem", "Rihanna", "Kanye West"]`). Без `$unwind` база згрупувала б дані за точним збігом усього масиву як єдиної сутності. Після `$unwind` кожен артист отримує цей трек у свій індивідуальний залік для підрахунку середнього чи мінімуму.
  * **Фільтрація вкладених структур:** Коли потрібно відсікти документи на основі властивостей конкретного елемента всередині масиву, вивівши в результат тільки цей елемент, а не весь масив повністю.

### Чим `$stdDevPop` відрізняється від `$stdDevSamp`?

Обидва оператори обчислюють **стандартне відхилення** (метрику того, наскільки сильно значення в наборі даних розкидані відносно їхнього середнього арифметичного), але вони базуються на різних статистичних моделях і формулах.

* **`$stdDevPop` (Population Standard Deviation — Стандартне відхилення генеральної сукупності):**
  * **Математика:** Розраховується за формулою, де знаменником є загальна кількість елементів $n$.
  * **Логіка:** Використовується тоді, коли набір даних є **повним і вичерпним** (тобто є абсолютно всі дані, які взагалі існують або нас цікавлять).
  * **Приклад:** У нашому завданні ми аналізували темп (`tempo`) треків у межах конкретного жанру. Оскільки ми рахували поріг аномалій для *всіх* наявних у нашій базі треків цього жанру, а не для випадкової вибірки, використання `$stdDevPop` є математично єдино правильним.

* **`$stdDevSamp` (Sample Standard Deviation — Стандартне відхилення вибірки):**
  * **Математика:** Використовує поправку Бесселя, де знаменником є $n - 1$ (що трохи збільшує фінальний результат, компенсуючи потенційну похибку).
  * **Логіка:** Використовується тоді, коли ми оперуємо лише **невеликою випадковою вибіркою** (семплом) з величезного масиву даних, і на основі цієї вибірки намагаємося спрогнозувати поведінку всієї генеральної сукупності.
  * **Приклад:** Якби у нас була база на 10 мільйонів треків Spotify, але для економії пам'яті ми взяли випадкові 500 треків і хотіли оцінити приблизний розкид темпу для всього стрімінгу загалом — тоді слід було б застосувати `$stdDevSamp`.


## Частина 3 — Аналітика через Aggregation Pipeline

### Зміна порогу кількості треків для виконавців (Запит 1)

**Контекст:** У Запиті 1 ми аналізуємо **середню популярність** треків виконавця.

* **Якщо знизити поріг до 1 треку:**
  * **Що зміниться:** Фінальний Топ-10 повністю деградує та заповниться так званими "ідолами одного хіта" (One-Hit Wonders) або штучно розкрученими синглами. У топ потраплять маловідомі артисти, які мають у базі всього 1–2 треки, але з випадково високим рейтингом популярності (наприклад, 90–95).
  * **Пояснення:** З погляду статистики, мала вибірка (1 трек) має колосальну дисперсію. Середнє значення по одному треку не відображає реальний статус артиста. Без жорсткого фільтра `total_tracks: { $gte: 5 }` великі системні артисти з десятками релізів (у яких середній бал природно розмивається менш успішними піснями чи інтро-треками з альбомів) будуть витіснені статистичним шумом.

* **Якщо підняти поріг до більш ніж 50 треків (`$gt: 50`):**
  * **Що зміниться:** Результат кардинально звузиться. З топу зникне більшість сучасних інді-виконавців, молодих поп-зірок та артистів, які випускають музику синглами або короткими EP (Extended Plays). У списку залишаться виключно "гіганти індустрії" з величезними дискографіями або старі класичні рок/поп-групи.
  * **Пояснення:** Такий високий поріг відсікає якісних виконавців, які просто фізично не встигли або не планують випускати таку велику кількість треків. Запит перестане показувати реальних лідерів за популярністю на сьогоднішній день і перетвориться на аналіз продуктивності ветеранів сцени.

### Зміна порогу кількості треків для музичних жанрів (Запит 3)

**Контекст:** У Запиті 3 ми аналізуємо **середні показники звуку** (`danceability`, `energy`, `valence`) у межах цілих музичних жанрів.

* **Чи зміниться результат, якщо знизити поріг зі 100 до 50 треків:**
  * **Коротка відповідь:** Результат зміниться мінімально або взагалі залишиться незмінним у верхній частині топу. 
  * **Пояснення:** На відміну від окремих людей (виконавців), музичний жанр за своєю природою є макро-категорією. Більшість репрезентативних жанрів у датасеті Spotify (рок, поп, техно, хіп-хоп) налічують сотні або тисячі треків. Зниження порогу до 50 дозволить пройти верифікацію лише дуже специфічним, локальним або рідкісним мікро-жанрам (наприклад, якийсь вузький піджанр фолку чи експериментального металу).
  * **Статистичний інсайт:** Для макро-агрегацій об'єм у 50 документів уже є цілком достатнім для спрацювання **центральної граничної теореми**. Середні значення (`$avg`) танцювальності чи енергії для великих жанрів при зміні цього порогу не зміняться, оскільки їхній об'єм значно вищий за обидва ліміти. Цей фільтр виконує роль технічного запобіжника від "сміттєвих" або помилково вказаних жанрів при імпорті CSV.


## Частина 4: Аналіз запиту та індексація

### Що змінилося в плані виконання?

План виконання запиту зазнав кардинальних архітектурних змін за трьома критичними метриками:

1. **Зміна стратегії сканування (Stage):** 

    * **Без індексу:** База даних виконувала етап `COLLSCAN` (Collection Scan). Вона була змушена повністю прочитати кожну сторінку пам'яті на диску, щоб знайти треки жанру "pop". Більше того, оскільки виконувалося сортування `.sort({ popularity: -1 })`, у плані з'являлася блокуюча стадія `SORT` — MongoDB виділяла оперативну пам'ять (RAM) і сортувала документи "на льоту". Якби об'єм даних перевищив 100 МБ, запит упав би з OOM-помилкою.

    * **З індексом:** План змінився на комбінацію `IXSCAN` (Index Scan) -> `FETCH`. Тепер база даних миттєво переходить до потрібної гілки B-Tree дерева індексу, вибирає лише релевантні вказівники та дістає з диска тільки цільові документи. Стадія `SORT` повністю зникла (`hasSortStage: false`), оскільки дані в індексі вже фізично лежать у відсортованому за спаданням популярності вигляді.

2. **Продуктивність (Ефективність сканування):**

   * **Без індексу:** Показник `totalKeysExamined` дорівнював `0` (індекси не чіпалися), а `totalDocsExamined` дорівнював загальній кількості документів у колекції (наприклад, 100 000+). Співвідношення повернутих документів до перевірених було катастрофічно низьким.
   
   * **З індексом:** `totalKeysExamined` чітко відповідає кількості документів, що потрапили під фільтр. Показник `totalDocsExamined` суворо дорівнює кількості документів, повернутих клієнту (`nReturned`). База даних не прочитала жодного зайвого документа з диска.

3. **Швидкість виконання (`executionTimeMillis`):**

   * Час виконання впав із сотень мілісекунд (або навіть секунд на великих базах) до `0-1 мс` (блискавичний відгук).

### Як зрозуміти, що індекс використовується?

Використання індексу залізобетонно підтверджується наступними ключовими полями з об'єкта `explain("executionStats")`:

* **`winningPlan.stage` або `executionStages.stage`:**
  Значення змінилося з `COLLSCAN` на `IXSCAN`. Це головний маркер того, що MongoDB пішла по шляху індексного дерева, а не лінійного перебору колекції.
  
* **`winningPlan.inputStage.indexName`:**
  Тут чітко відображається системне ім'я створеного нами індексу, наприклад: `track_genre_1_popularity_-1_audio_features.danceability_1`.

* **Порівняння метрик сканування:**
  * `totalKeysExamined: <число_більше_0>` — показує, скільки індексних ключів було прочитано.
  * `totalDocsExamined` — його значення тепер мінімальне і суворо обмежене лише тими документами, які дійсно відповідають критеріям пошуку.

#### Приклад значень із блоку `explain()`:

```json
"executionStats" : {
    "executionSuccess" : true,
    "nReturned" : 1245, // кількість знайдених поп-треків
    "executionTimeMillis" : 1, // швидкість на рівні похибки
    "totalKeysExamined" : 1245, // прочитано рівно стільки ключів, скільки треба
    "totalDocsExamined" : 1245, // нуль зайвих читань з диска
    "executionStages" : {
        "stage" : "FETCH",
        "nReturned" : 1245,
        "inputStage" : {
            "stage" : "IXSCAN",
            "indexName" : "track_genre_1_popularity_-1_audio_features.danceability_1",
            "isMultiKey" : false,
            "direction" : "forward"
        }
    }
}
```

### Складений індекс для фонової музики

Для оптимізації частих запитів пошуку інструментальної, тихої музики без нецензурного контенту було створено складений індекс за правилом **ESR**:
`{ explicit: 1, "audio_features.instrumentalness": 1, "audio_features.speechiness": 1 }`

#### Докази використання індексу з плану виконання (`explain`)

Успішна активація індексу та перехід бази даних від лінійного сканування до деревоподібного пошуку підтверджується такими ключовими параметрами об'єкта `executionStats`:

1. **Зміна стадії (Stage):**
   У полі `winningPlan.queryPlan.stage` (або всередині `executionStages`) замість важкого `COLLSCAN` (сканування всієї колекції) чітко зафіксовано стадію **`IXSCAN` (Index Scan)**. Це означає, що рушій WiredTiger знайшов потрібні документи через B-Tree дерево індексу.

2. **Ім'я активованого індексу:**
   У полі `indexName` відображається системний ідентифікатор створеного нами індексу:
   `"indexName" : "track_genre_1_popularity_-1_audio_features.danceability_1'"`

3. **Точність сканування:**
   Показник `totalDocsExamined` (кількість документів, які база фізично прочитала з диска) тепер суворо обмежений лише тими записами, які відповідають усім критеріям фільтра. База даних більше не витрачає ресурси CPU та дискового I/O на перевірку треків з високим вмістом мови або explicit-контентом.

### Аналіз покривного запиту (Covered Query)

**Дано запит:**
```javascript
db.tracks.find({
  track_genre: "pop",
  popularity: { $gte: 70 }
});
```
індекс:
```javascript
{ track_genre: 1, popularity: -1, "audio_features.danceability": 1 }
```

Ні, цей запит не є покривним.
Щоб запит у MongoDB став покривним (Covered Query), мають одночасно виконуватися дві фундаментальні умови:

1. Усі поля, які використовуються у фільтрації (find), повинні бути частиною активованого індексу. (Ця умова виконується, оскільки track_genre та popularity є в нашому складеному індексі).

2. Критична умова: Усі поля, які повертаються клієнту у вибірці (проєкція), повинні міститися виключно всередині самого індексу. База даних не повинна звертатися до самого документа на диску.

Ось покривний запит:
```javascript
db.tracks.find({
  track_genre: "pop",
  popularity: { $gte: 70 }
}, {
  _id: 0,              // КРИТИЧНО: вимикаємо _id, якого немає в індексі
  track_genre: 1,      // беремо прямо з індексу
  popularity: 1        // беремо прямо з індексу
});
```

Якщо запустити .explain("executionStats") для обох варіантів запиту, різниця у плані виконання чітко покаже правду:

1. Для початкового (непокривного) запиту:

    У структурі етапів буде ланцюжок: IXSCAN -> FETCH.

    Показник totalDocsExamined буде більшим за 0 (він буде дорівнювати кількості знайдених документів, бо базу змусили читати їх з диска).

2. Для оптимізованого (покривного) запиту:

    Стадія FETCH повністю зникне. Залишиться тільки IXSCAN.

    Показник totalDocsExamined буде суворо дорівнювати 0. Це головний технічний доказ того, що запит є покривним. Рушій бази даних видав результат, взагалі не доторкнувшись до самої колекції на диску, провівши всю операцію суто в пам'яті індексу.
