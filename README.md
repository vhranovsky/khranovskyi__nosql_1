# Домашнє завдання: Трансформація даних Spotify у MongoDB

Цей проєкт містить скрипт для очищення, денормалізації та структурування сирих даних музичних треків у базі даних MongoDB, а також архітектурне обгрунтування обраних рішень.

## 1. Скрипт трансформації даних (`02_transform.js`)

Скрипт написаний для середовища `mongosh`. Він повністю автоматизує процес очищення даних, створення вкладених об'єктів, обчислення нових метрик та збереження фінальної структури.

## 2. Відповіді на теоретичні питання
### Питання 1: Чому аудіо-характеристики винесені в окремий об’єкт audio_features, а не зберігаються плоско? Коли таке вкладення вигідне, а коли створює проблеми?
Обгрунтування: Це патерн логічного групування (Logical Grouping). Аудіо-фічі (темп, акустичність, енергійність) описують суто фізичні властивості звукової доріжки і суттєво відрізняються від загальних метаданих треку (назва, жанр, альбом).

Коли це вигідно:
Оптимізація вибірки (Проєкція): Якщо для інтерфейсу застосунку потрібні лише базові дані, можна легко виключити весь блок фіч за допомогою { audio_features: 0 }, зменшуючи навантаження на мережу та RAM.

Зручність для аналітики/ML: При передачі даних у моделі рекомендаційних систем зручніше оперувати одним готовим об'єктом, ніж збирати 12 розрізнених полів по всьому документу.

Коли це створює проблеми:
Громіздкі запити: Для пошуку та індексації доводиться постійно писати довгі шляхи через крапкову нотацію, наприклад: db.tracks.find({ "audio_features.tempo": { $gt: 120 } }).

Ризик затирання даних: При оновленні вкладеного об'єкта без використання точкового оператора $set можна випадково видалити інші характеристики, що містилися всередині об'єкта.

### Питання 2: Чому виконавці зберігаються як масив, а не як рядок? Які запити стають простішими?
Обгрунтування: Рядок типу "Artist A; Artist B" порушує першу нормальну форму (1NF), оскільки дані в межах одного поля не є атомарними. Масив — це природний NoSQL-спосіб реалізації зв'язків один-до-багатьох (1:N).

Які запити стают
Точний пошук: Пошук треків конкретного виконавця виконується простим запитом db.tracks.find({ artists: "Eminem" }). Немає потреби у важких регулярних виразах ($regex), які змушують базу робити повне сканування колекції (Collection Scan).

Пошук спільних треків (коллаборацій): Запити на перетин множин стають тривіальними завдяки оператору $all: db.tracks.find({ artists: { $all: ["Jay-Z", "Linkin Park"] } }).

Індексація (Multikey Index): MongoDB дозволяє побудувати індекс по масиву. База створить окремий покажчик для кожного артиста, що робить пошук за іменем миттєвим. З текстовим рядком така індексація працювала б набагато повільніше та менш ефективно.

### Питання 3: Що таке $out і чим він відрізняється від $merge? Коли використовувати кожен?
Обидва оператори використовуються на фінальній стадії конвеєра агрегації (Aggregation Pipeline) для збереження результатів, але вони мають кардинально різну логіку роботи під капотом:

$out (Повний перезапис):
Механіка: Повністю знищує (робить drop) цільову колекцію, якщо вона існувала, і створює її з нуля, записуючи туди нові документи.

Коли використовувати: Ідеально підходить для пакетної обробки (Batch Processing), перегенерації щомісячних/щоденних звітів, матеріалізованих представлень, або під час первинної міграції та повної трансформації даних (як у поточному завданні).

$merge (Інкрементальне оновлення):
Механіка: Не видаляє існуючу колекцію. Він зіставляє документи з результату агрегації з документами в цільовій колекції за унікальним ключем (наприклад, _id). Залежно від умов, він може оновити старий документ, додати новий (якщо збігів не знайдено) або виконати гнучку логіку злиття полів.

Коли використовувати: Незамінний для обробки даних у реальному часі. Наприклад, коли потрібно додати нові щоденні прослуховування до вже існуючої історичної бази статистики за рік, не перераховуючи при цьому дані за попередні 11 місяців.

## Для чого використовується інструкція `$unwind`?
**Обгрунтування та механіка:**
Інструкція `$unwind` використовується для денормалізації(розгортання) масивів у документах під час виконання конвеєра агрегації(Aggregation Pipeline). 

Коли MongoDB зустрічає поле, яке є масивом, вона не може застосувати до його елементів стандартні операції групування або фільтрації так, ніби це окремі документи. Етап `$unwind` бере один вихідний документ, що містить масив із `N` елементів, і розбиває його на `N` окремих копій-документів. У кожній такій копії на місці колишнього масиву тепер міститься лише один конкретний його елемент.

* **Коли це критично необхідно:**
  * **Групування та аналітика:** Як у завданні з артистами, де один трек міг належати масиву з трьох виконавців (`["Eminem", "Rihanna", "Kanye West"]`). Без `$unwind` база згрупувала б дані за точним збігом усього масиву як єдиної сутності. Після `$unwind` кожен артист отримує цей трек у свій індивідуальний залік для підрахунку середнього чи мінімуму.
  * **Фільтрація вкладених структур:** Коли потрібно відсікти документи на основі властивостей конкретного елемента всередині масиву, вивівши в результат тільки цей елемент, а не весь масив повністю.

## Чим `$stdDevPop` відрізняється від `$stdDevSamp`?

Обидва оператори обчислюють **стандартне відхилення** (метрику того, наскільки сильно значення в наборі даних розкидані відносно їхнього середнього арифметичного), але вони базуються на різних статистичних моделях і формулах.

* **`$stdDevPop` (Population Standard Deviation — Стандартне відхилення генеральної сукупності):**
  * **Математика:** Розраховується за формулою, де знаменником є загальна кількість елементів $n$.
  * **Логіка:** Використовується тоді, коли набір даних є **повним і вичерпним** (тобто є абсолютно всі дані, які взагалі існують або нас цікавлять).
  * **Приклад:** У нашому завданні ми аналізували темп (`tempo`) треків у межах конкретного жанру. Оскільки ми рахували поріг аномалій для *всіх* наявних у нашій базі треків цього жанру, а не для випадкової вибірки, використання `$stdDevPop` є математично єдино правильним.

* **`$stdDevSamp` (Sample Standard Deviation — Стандартне відхилення вибірки):**
  * **Математика:** Використовує поправку Бесселя, де знаменником є $n - 1$ (що трохи збільшує фінальний результат, компенсуючи потенційну похибку).
  * **Логіка:** Використовується тоді, коли ми оперуємо лише **невеликою випадковою вибіркою** (семплом) з величезного масиву даних, і на основі цієї вибірки намагаємося спрогнозувати поведінку всієї генеральної сукупності.
  * **Приклад:** Якби у нас була база на 10 мільйонів треків Spotify, але для економії пам'яті ми взяли випадкові 500 треків і хотіли оцінити приблизний розкид темпу для всього стрімінгу загалом — тоді слід було б застосувати `$stdDevSamp`.
